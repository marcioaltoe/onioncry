# Use features as the default Vertical Slice root

OnionCry's Vertical Slice organization guidance will prefer `src/features/<feature>` as the default slice layout. This matches common TypeScript feature-folder practice while keeping the source root free for base folders such as `app`, `config`, `lib`, `shared`, or entrypoint files. Projects may configure another `sliceRoot`, such as `slices`, `modules`, or `.`, but root-level slices are explicit because they are more ambiguous to classify. A slice's default public surface is `index.ts` and `contracts/`; internal folders such as `handlers/`, `adapters/`, `domain/`, and `__tests__/` are validated when present but are not required as empty scaffolding. Filename suffixes such as `.repository.ts`, `.service.ts`, `.handler.ts`, `.adapter.ts`, `.entity.ts`, `.value-object.ts`, and `.use-case.ts` may be used as configurable artifact classifiers alongside folder placement. In Vertical Slice mode, `.service.ts` files are slice-internal by default and must not be imported by other slices unless exposed through the configured public surface. Vertical Slice mode does not run Clean Architecture checks and does not automatically reject global `domain`, `application`, or `infra` folders; a separate `verticalslice/no-global-slice-artifacts` rule can warn when slice-looking artifacts drift outside the configured slice root while configured global folders remain allowed.
