# Use features as the default Vertical Slice root

OnionCry's Vertical Slice organization guidance will prefer `src/features/<domain>/<operation>` as the default slice layout. This keeps the source root free for base folders such as `app`, `config`, `lib`, `shared`, or entrypoint files while giving larger projects one grouping level for a domain and one level for an independently deliverable operation. Projects may configure another `sliceRoot`, such as `slices`, `modules`, or `.`, and may set `sliceDepth: 1` when they intentionally use `features/<feature>` or root-level feature folders. A slice's default public surface is `index.ts` and `contracts/`; internal folders such as `handlers/`, `adapters/`, `domain/`, and `__tests__/` are validated when present but are not required as empty scaffolding. Filename suffixes such as `.repository.ts`, `.service.ts`, `.handler.ts`, `.adapter.ts`, `.entity.ts`, `.value-object.ts`, and `.use-case.ts` may be used as configurable artifact classifiers alongside folder placement. In Vertical Slice mode, `.service.ts` and `.repository.ts` files are slice-internal by default and must not be imported by other slices unless exposed through the configured public surface. Vertical Slice mode does not run Clean Architecture checks and does not automatically reject global `domain`, `application`, or `infra` folders; `verticalslice/no-global-slice-artifacts` can warn when slice-looking artifacts drift outside the configured slice root while configured global folders remain allowed, `verticalslice/no-shared-layer-artifacts` can report shared technical layers outside slices, and `verticalslice/slice-entry-point` can require slices to expose a configured entry point such as `setup`, `Map`, or `register`.
