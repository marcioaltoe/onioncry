# Use context-first Clean Architecture layout

OnionCry's Clean Architecture organization guidance will prefer `src/contexts/<context>/{domain,application,infra}` for contextual backend code and `src/{domain,application,infra}` for contextless base code. This keeps the Clean Architecture dependency direction while avoiding long global folders such as `application/use-cases`, `application/ports`, `infra/repositories`, and `infra/adapters`. The defaults remain configurable under `architecture.cleanArchitecture`: projects may use another context root segment such as `modules`, may map canonical layer names to path aliases such as `infra` -> `infrastructure`, and may configure artifact folders per canonical layer. The structure rules are presence-based: OnionCry validates where use cases, ports, repositories, adapters, entities, and value objects belong when they exist, but it does not require empty layer folders in every context. Filename suffixes can help classify artifacts, but ambiguous suffixes such as `.service.ts` are interpreted through their containing layer or artifact folder. The `cleanarch/artifact-placement` rule defaults to `warn` so existing projects can expose structural drift before turning it into a blocking check.
